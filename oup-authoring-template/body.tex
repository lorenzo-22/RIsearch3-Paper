\section{Introduction}

Intermolecular RNA--RNA interactions underpin a wide range of regulatory
mechanisms, and predicting them at genome- or transcriptome-wide scale is a
recurring task in RNA biology. The problem is particularly demanding for
synthetic small interfering RNAs (siRNAs), where unintended off-target
binding can confound RNAi knockdown experiments and limit therapeutic
application.

RIsearch2~\citep{alkan2017risearch2} is a widely used tool for large-scale
screens of near-complementary RNA--RNA interactions: it combines a
suffix-array index of the target with a seed-and-extend dynamic-programming
step that uses the simplified energy model introduced by
RIsearch~\citep{wenzel2012risearch}. The companion siRNA off-target pipeline
distributed alongside RIsearch2 turns these predictions into per-transcript
binding probabilities and a transcriptome-wide off-targeting potential
through a partition-function model that integrates transcript abundance and
binding-site accessibility.

Despite the algorithm's continued relevance, the software ecosystem has aged
less well: RIsearch2 ships as a C source tarball with a custom Makefile; the
off-target pipeline is written in Python~2.7, long past end of life; and
neither component exposes a library API suitable for embedding in modern
Python-native analyses.

We present \textbf{RIsearch3}: a Rust reimplementation of the RIsearch2
algorithm with parity-tested behaviour, a redesigned command-line interface,
and PyO3 Python bindings that expose search results as native Polars
DataFrames. Alongside the core, we release a rewritten siRNA off-target
pipeline that
consumes the bindings directly and thus doubles as a reference example for
users wishing to embed RIsearch3 in their own pipelines.

\section{siRNA Off-Target Discovery Pipeline}\label{sec:pipeline}

The new siRNA off-target discovery pipeline is a full reimplementation of the
workflow described in \citet{alkan2017risearch2}, ported from Python~2.7 to
Python~3.14, restructured as a modular package, and accelerated through
vectorised DataFrame operations and multi-core parallelism.

\subsection*{Invocation and configuration}

The pipeline exposes two subcommands: \texttt{off-targets}, which executes
the full analysis, and \texttt{accessibility}, which pre-computes per-site
RNA opening-energy profiles from a genome or transcriptome FASTA using the
RNAplfold algorithm (ViennaRNA~2.7.2 Python API~\citep{lorenz2011vienna};
default $W{=}80$, $L{=}40$, $u{=}30$).
Both subcommands accept a \texttt{-{}-temperature} parameter (default
$37\,^\circ$C) that is propagated to all thermodynamic calculations: in
\texttt{accessibility}, it sets the ViennaRNA model temperature
(\texttt{md.temperature}), so that opening energies $O_i = -RT\ln p_i$ are
computed at the specified condition; in \texttt{off-targets}, the same
temperature determines the thermal energy $RT$ in the Boltzmann partition
function.
Sharing a single temperature across both stages ensures that hybridisation
energies $E^\prime_i$ from RIsearch and opening energies $O_i$
from ViennaRNA are evaluated on a consistent thermodynamic scale before
summation, directly replicating the intent of the original RIsearch
model~\citep{wenzel2012risearch}.
All remaining parameters---RIsearch energy threshold, seed specification,
saturation parameters $\alpha$, $\gamma$, and $\theta$, and paths to query,
transcriptome, and on-target files---are declared in a single YAML
configuration file validated at startup against a typed OmegaConf schema.
Individual subcommands may be invoked independently to support staged
workflows.

\subsection*{Pipeline stages}

The four-stage model of \citet{alkan2017risearch2} is preserved
(Figure~\ref{fig:workflow}). In the first
stage, siRNA--target interaction predictions are generated by calling
\texttt{risearch.index()} to build the suffix-array index of the target
transcriptome and \texttt{risearch.search()} to locate binding sites, both
invoked directly through the Python bindings
(Section~\ref{subsec:pybindings}), returning results as a Polars DataFrame
in the same process with no intermediate files on disk. In the second stage,
predicted binding sites are intersected with a transcriptome annotation (GTF
or BED) to retain sites overlapping expressed features and to assign
position-specific abundance values. Intersection is performed in-memory using
an NCLS interval index~\citep{ncls} partitioned by chromosome and strand, with
chromosome--strand pairs distributed across a thread pool; no external tools
such as \texttt{bedtools} are needed. In the third stage, per-site binding-site opening energies $O_i = -RT\ln p_i$
are retrieved from per-chromosome Parquet profiles pre-computed by the
\texttt{accessibility} subcommand using the RNAplfold algorithm
(ViennaRNA~2.7.2 Python API; $W{=}80$, $L{=}40$, $u{=}30$) at the same
temperature used for the partition function, so that $E^\prime_i$
and $O_i$ are on a consistent thermodynamic scale; the Boltzmann-weighted sum
%
\begin{equation}
  Z_s = \sum_{i \in \mathcal{I}_s}
        A_i \,
        \exp\!\left(\frac{-\bigl(E^\prime_i + O_i\bigr)}{RT}\right)
  \label{eq:partition}
\end{equation}
%
\noindent is evaluated as a single vectorised \texttt{group\_by} aggregation
in Polars, where $A_i$ is the transcript expression level,
$E^\prime_i$ is the RIsearch hybridisation energy,
$O_i = -RT\ln p_i$ is the opening energy derived from the RNAplfold unpaired
probability $p_i$, $R = 1.987\times10^{-3}$\,kcal\,mol$^{-1}$\,K$^{-1}$ is
the molar gas constant, and $T$ is the folding temperature (default
$310.15$\,K, $37\,^\circ$C). When an on-target binding site is supplied via \texttt{-{}-on-target-ids},
its Boltzmann weight $W_{\mathrm{on}}$ is computed identically and added to
$Z_s$; the on-target probability
$p_{\mathrm{on}} = W_{\mathrm{on}} / Z_s$
determines the aggregate off-targeting potential
$P_{\mathrm{OFF}} = 1 - p_{\mathrm{on}}$. Per-transcript
off-targeting probabilities $p_{\mathrm{off},i} = W_i / Z_s$ are then computed as column
operations on the resulting DataFrame. Parametric sweeps over $\alpha$,
$\gamma$, and $\theta$ are resolved in a single aggregation pass rather than
repeated sequential iterations. In the fourth stage, results are written
incrementally as each siRNA completes, in TSV, CSV, or Parquet format.

\subsection*{Parallelism and memory}

When processing a directory of per-siRNA prediction files, the pipeline
distributes work across a configurable process pool. A spawn-based
multiprocessing context is used to avoid interaction with the Rayon thread
pool used internally by Polars. The transcriptome annotation is loaded once
into Arrow IPC format and memory-mapped into each worker process via the
operating-system page cache, keeping peak memory proportional to a single
transcriptome copy regardless of worker count. Prediction files are parsed
lazily so that only filtered, selected columns are materialised; output is
streamed to disk incrementally throughout a batch run.

Benchmarked on an AMD EPYC 9554 (64 cores, 32 workers) using subsets of
the genome-wide siRNA efficacy library of \citet{huesken2005design} searched
against the repeat-masked GRCh38/hg38 reference genome and intersected against
an H1299 RNA-seq expression profile (Supplementary Methods), the pipeline
completes the intersection, accessibility annotation, probability computation,
and output stages $7{-}9\times$ faster than the legacy Python~2.7
implementation across batch sizes of 500--2{,}000 siRNAs (five randomised
trials per batch size; Supplementary Figure~S1). Peak memory across all
workers plateaus at approximately 7\,GB in the new pipeline regardless of
batch size, whereas the legacy pipeline grows from $\sim$8.5\,GB to
$\sim$10\,GB as batch size increases.
